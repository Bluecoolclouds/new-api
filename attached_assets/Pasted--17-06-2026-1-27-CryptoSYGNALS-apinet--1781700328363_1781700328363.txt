Что поддерживает apinet

Сейчас в коде есть несколько разных механизмов web search, в зависимости от провайдера/канала:

1) OpenAI-style chat completions

В общий request уже добавлено поле:

• web_search_options
• плюс vendor-specific поля:
  • search_parameters — для xAI
  • enable_search — для Qwen/Ali-подобных
  • web_search — для Baidu v2
  • search_domain_filter, search_recency_filter, search_mode — для Perplexity

Это видно в:

• /root/apinet/dto/openai_request.go

───

2) Claude web search

Поддержка есть явно.

Если в запросе приходит:

"web_search_options": {
  "search_context_size": "low|medium|high",
  "user_location": {...}
}

то apinet превращает это в Claude tool:

• type: "web_search_20250305"
• name: "web_search"

и мапит:

• low → max_uses = 1
• medium → max_uses = 5
• high → max_uses = 10

Файлы:

• /root/apinet/relay/channel/claude/relay-claude.go
• /root/apinet/dto/claude.go

И ещё считает usage:

• claude_web_search_requests

───

3) OpenAI Responses API built-in web search

Поддержка тоже есть.

Код знает built-in tool:

• web_search_preview

и built-in call:

• web_search_call

То есть если использовать Responses API и передать tool типа:

{
  "type": "web_search_preview",
  "search_context_size": "medium"
}

apinet:

• сохраняет этот tool в relay info
• считает вызовы web_search_call
• умеет биллить это как tool usage

Файлы:

• /root/apinet/dto/openai_response.go
• /root/apinet/relay/common/relay_info.go
• /root/apinet/relay/channel/openai/relay_responses.go

───

4) Perplexity

Есть отдельный канал/adaptor perplexity.

Прокидываются поля:

• search_domain_filter
• search_recency_filter
• return_images
• return_related_questions
• search_mode

Файлы:

• /root/apinet/relay/channel/perplexity/*

То есть для Perplexity search — это нативная часть канала, не просто prompt.

───

5) xAI / Grok

Есть поддержка через суффикс модели -search.

Если upstream model заканчивается на -search, то adaptor:

• убирает суффикс из model
• автоматически добавляет в request:

"search_parameters": {
  "mode": "on"
}

Файл:

• /root/apinet/relay/channel/xai/adaptor.go

───

6) Baidu v2

Есть похожая логика через суффикс модели -search.

Если модель оканчивается на -search, то автоматически добавляется:

"web_search": {
  "enable": true,
  "enable_citation": true,
  "enable_trace": true,
  "enable_status": false
}

Файл:

• /root/apinet/relay/channel/baidu_v2/adaptor.go

───

7) Qwen / Ali

Есть следы поддержки internet-search режима:

• константа -internet
• поле enable_search

Но по коду, который я посмотрела, полной автоконверсии как у xAI/Baidu я не вижу в этом фрагменте. То есть DTO под это готово, но надо ещё отдельно смотреть, где конкретно enable_search реально выставляется.

Как это реализовано в apinet

Для chat completions

apinet принимает расширенные поля прямо в общем OpenAI-compatible request:

{
  "model": "...",
  "messages": [...],
  "web_search_options": {
    "search_context_size": "medium"
  }
}

Дальше в TextHelper это проходит через adaptor конкретного провайдера.

Файл:

• /root/apinet/relay/compatible_handler.go

Там ещё есть валидация:

• search_context_size должен быть low | medium | high
• если не указан, ставится medium

Файл:

• /root/apinet/relay/helper/valid_request.go

───

Для OpenAI Responses API

Search идёт не через web_search_options, а через built-in tools.

То есть запрос должен быть примерно такого типа:

{
  "model": "gpt-...",
  "input": "...",
  "tools": [
    {
      "type": "web_search_preview",
      "search_context_size": "medium"
    }
  ]
}

И apinet уже умеет:

• распознать этот tool
• отследить вызовы web_search_call
• посчитать usage/price

Что по сути “какие модели поддерживают”

Вот важный момент: в apinet нет одного централизованного списка “эти модели умеют web search”.

Поддержка сейчас выглядит так:

• Claude — да, через web_search_options
• OpenAI Responses-compatible модели — да, если модель/канал реально умеет built-in web_search_preview
• Perplexity — да, нативно
• xAI — да, через -search / search_parameters.mode=on
• Baidu v2 — да, через -search / web_search.enable=true
• Qwen/Ali — похоже частично подготовлено, но надо дочитать конкретный adaptor
• Gemini — явной законченной реализации в этом слое я не увидела

То есть ответ: это не “у модели само по себе”, а завязано на канал/adaptor и формат запроса.

Web Search в apinet: как использовать

Коротко

В apinet web search не включается одинаково для всех моделей.

Есть 3 основных варианта:

1. через web_search_options
подходит для части OpenAI-compatible запросов, особенно для Claude-интеграции

2. через Responses API tools
для моделей/каналов, которые поддерживают built-in tool web_search_preview

3. через provider-specific параметры
например:

  • xAI / Grok → search_parameters
  • Perplexity → search-поля в запросе
  • Baidu → web_search
  • Qwen/Ali → enable_search (если канал это реально поддерживает)

───

Главное правило

Просто написать в prompt:

“поищи в интернете”

недостаточно.

Если search не включён через request body или не поддержан конкретным каналом/моделью, модель либо:

• не сможет искать,
• либо начнёт фантазировать.

───

Вариант 1. Claude / OpenAI-compatible chat completions

Если ваш клиент работает через обычный endpoint в стиле:

• /v1/chat/completions

можно передавать:

{
  "model": "claude-...",
  "messages": [
    {
      "role": "user",
      "content": "Какие новости по OpenAI за сегодня?"
    }
  ],
  "web_search_options": {
    "search_context_size": "medium"
  }
}

Параметры

web_search_options.search_context_size

Допустимые значения:

• low
• medium
• high

Если не указать, по коду apinet по умолчанию ставит:

"medium"

Когда использовать

Используйте этот способ, если:

• у вас chat completions
• модель/канал поддерживает web search через этот механизм
• особенно если это Claude

Вариант 2. OpenAI Responses API

Если вы используете endpoint в стиле:

• /v1/responses

то search включается через tools.

Пример:

{
  "model": "gpt-4.1",
  "input": "Найди свежую информацию о релизах OpenAI за последние 7 дней",
  "tools": [
    {
      "type": "web_search_preview",
      "search_context_size": "medium"
    }
  ]
}

Когда использовать

Используйте этот способ, если:

• ваш клиент уже работает с Responses API
• модель/канал поддерживает built-in tool web_search_preview

Важно

Для Responses API search включается не через web_search_options, а именно через:

"tools": [{ "type": "web_search_preview", ... }]

Для xAI в apinet есть специальная логика: search включается через search-параметры.

Условно:

• обычная модель: grok-...
• search-вариант: grok-...-search

Как выбрать search_context_size

low

Когда нужен:

• быстрый ответ
• минимальный search overhead
• грубая проверка фактов

medium

Лучший default.
Подходит почти для всего:

• новости
• цены
• актуальные факты
• быстрые сравнительные ответы

high

Используйте когда:

• нужен более широкий поиск
• важна полнота
• запрос сложный или исследовательский
